Secuencias
Oracle no tiene AUTOINCREMENT. Cada PK numérica necesita su propia secuencia.

En Oracle se crean con CREATE SEQUENCE y se usan en los INSERT con nombre_seq.NEXTVAL. Opcionalmente se pueden asociar a columnas con DEFAULT en Oracle 12c+.

Regla: Una secuencia por cada tabla que tenga PK numérica autogenerada. Las tablas con PK compuesta o PK varchar no necesitan secuencia.

03
Constraints
Las reglas de negocio del enunciado deben reforzarse desde la base de datos.

No basta con tener PKs y FKs. El enunciado tiene reglas de negocio explícitas que deben vivir en la BD como CHECK constraints, de forma que ninguna aplicación pueda violarlas.

CHECK constraints obligatorios Reglas de negocio
-- Modalidad de cobro: solo dos valores posibles CONSTRAINT chk_modalidad CHECK (modalidad IN ('GLOBAL', 'CREDITOS')) -- Grupo de código de detalle: COBRO o PAGO CONSTRAINT chk_grupo CHECK (grupo IN ('COBRO', 'PAGO')) -- Estado del volante: valores controlados CONSTRAINT chk_estado_volante CHECK (estado IN ('PENDIENTE', 'PAGADO', 'PARCIAL')) -- Tipo de generación del volante CONSTRAINT chk_tipo_gen CHECK (tipo_generacion IN ('INDIVIDUAL', 'MASIVA')) -- Valor del movimiento no puede ser negativo CONSTRAINT chk_valor_positivo CHECK (valor > 0) -- Semestre entre 1 y 10 CONSTRAINT chk_semestre CHECK (semestre BETWEEN 1 AND 10)
UNIQUE constraints importantes Integridad
-- Un estudiante solo puede tener una cuenta corriente CONSTRAINT uk_cuenta_estudiante UNIQUE (id_estudiante) -- en tabla cuenta_corriente -- Un usuario solo puede tener una persona asociada CONSTRAINT uk_usuario_persona UNIQUE (cedula) -- en tabla usuario -- Un movimiento solo puede asociarse a una transaccion CONSTRAINT uk_transaccion_movimiento UNIQUE (id_mov) -- en tabla transaccion_pago -- Username único en el sistema CONSTRAINT uk_username UNIQUE (username) -- en tabla usuario
ON DELETE en FK — decisiones importantes Cuidado
-- Si se borra un volante, sus inscripciones de asignatura se borran CONSTRAINT fk_vma_volante FOREIGN KEY (id_volante) REFERENCES volante_matricula(id_volante) ON DELETE CASCADE -- NO borrar movimientos si se borra una cuenta (proteger historial) CONSTRAINT fk_mov_cuenta FOREIGN KEY (id_cuenta) REFERENCES cuenta_corriente(id_cuenta) -- Sin ON DELETE: Oracle lanzará error si se intenta borrar

Índices
Columnas que se filtran frecuentemente en las consultas y reportes del enunciado.

Oracle crea automáticamente índices para PK y UNIQUE. Los índices adicionales deben crearse sobre columnas FK que se usan como filtro en JOINs y WHERE frecuentes. Los reportes del enunciado definen cuáles son críticos.

05
Vistas
Sí son necesarias. Simplifican las consultas de la web app y encapsulan los JOINs complejos.
✅ Las vistas son la interfaz entre la BD y la aplicación web. En lugar de que el código construya JOINs de 5 tablas cada vez, consulta la vista y obtiene exactamente lo que necesita mostrar.
v_cuenta_corriente_detalle Vista principal

Usada en la pantalla de cuenta corriente del estudiante. Trae movimientos con saldo acumulado por período.

CREATE OR REPLACE VIEW v_cuenta_corriente_detalle AS SELECT e.id_estudiante, e.nombre || ' ' || e.apellido AS nombre_completo, m.id_mov, m.fecha, cd.grupo, cd.codigo_Detalle, cd.descripcion AS desc_movimiento, m.valor, p.id_periodo, p.nombre_periodo, SUM( CASE WHEN cd.grupo = 'COBRO' THEN m.valor WHEN cd.grupo = 'PAGO' THEN -m.valor END ) OVER ( PARTITION BY cc.id_cuenta, p.id_periodo ORDER BY m.fecha, m.id_mov ) AS saldo_acumulado FROM movimiento m JOIN cuenta_corriente cc ON m.id_cuenta = cc.id_cuenta JOIN estudiante e ON cc.id_estudiante = e.id_estudiante JOIN codigo_detalle cd ON m.codigo_detalle = cd.codigo_Detalle JOIN periodo_academico p ON m.id_periodo = p.id_periodo;
v_volante_completo Vista de volante

Usada al mostrar o imprimir un volante de matrícula. Consolida estudiante, programa, período y regla de cobro.

CREATE OR REPLACE VIEW v_volante_completo AS SELECT v.id_volante, e.id_estudiante, e.nombre || ' ' || e.apellido AS estudiante, e.carnet, p.nombre_programa, per.nombre_periodo, v.semestre_que_cobra, v.modalidad, v.estado, v.fecha_generacion, v.tipo_generacion, rc.valorGlobal, rc.valorCredito, v.monto_total FROM volante_matricula v JOIN estudiante e ON v.id_estudiante = e.id_estudiante JOIN programa_academico p ON v.id_programa = p.id_programa JOIN periodo_academico per ON v.id_periodo = per.id_periodo JOIN regla_cobro rc ON (v.id_programa = rc.id_programa AND v.id_periodo = rc.id_periodo AND v.modalidad = rc.modalidad);
v_saldo_periodo Vista de saldo

Usada para mostrar el balance COBROs - PAGOs = 0 por estudiante y período. Es la vista del requisito central del enunciado.

CREATE OR REPLACE VIEW v_saldo_periodo AS SELECT e.id_estudiante, e.nombre || ' ' || e.apellido AS estudiante, p.id_periodo, p.nombre_periodo, SUM(CASE WHEN cd.grupo='COBRO' THEN m.valor ELSE 0 END) AS total_cobros, SUM(CASE WHEN cd.grupo='PAGO' THEN m.valor ELSE 0 END) AS total_pagos, SUM(CASE WHEN cd.grupo='COBRO' THEN m.valor WHEN cd.grupo='PAGO' THEN -m.valor END) AS saldo_neto FROM movimiento m JOIN cuenta_corriente cc ON m.id_cuenta = cc.id_cuenta JOIN estudiante e ON cc.id_estudiante = e.id_estudiante JOIN codigo_detalle cd ON m.codigo_detalle = cd.codigo_Detalle JOIN periodo_academico p ON m.id_periodo = p.id_periodo GROUP BY e.id_estudiante, e.nombre, e.apellido, p.id_periodo, p.nombre_periodo;
06
Triggers
Las reglas de negocio más críticas del enunciado deben vivir en triggers.
Principio: Todo lo que el enunciado dice que "debe suceder automáticamente" es un trigger. Si depende de que la aplicación lo recuerde hacer, es frágil.
trg_crear_cuenta_corriente CRÍTICO

El enunciado dice explícitamente: "Crear la cuenta corriente de un estudiante nuevo al momento de generar el cobro ya sea en forma individual o masiva." Esto es un trigger AFTER INSERT en volante_matricula.

CREATE OR REPLACE TRIGGER trg_crear_cuenta_corriente AFTER INSERT ON volante_matricula FOR EACH ROW DECLARE v_existe NUMBER; BEGIN SELECT COUNT(*) INTO v_existe FROM cuenta_corriente WHERE id_estudiante = :NEW.id_estudiante; IF v_existe = 0 THEN INSERT INTO cuenta_corriente (id_cuenta, id_estudiante) VALUES (seq_cuenta.NEXTVAL, :NEW.id_estudiante); END IF; END;
trg_calcular_monto_total Importante

Cuando se insertan asignaturas en volante_matricula_asignatura (modalidad CREDITOS), el monto_total del volante debe recalcularse automáticamente.

CREATE OR REPLACE TRIGGER trg_calcular_monto_total AFTER INSERT OR DELETE ON volante_matricula_asignatura FOR EACH ROW DECLARE v_id_volante NUMBER; v_total NUMBER(15,2); BEGIN v_id_volante := CASE WHEN INSERTING THEN :NEW.id_volante ELSE :OLD.id_volante END; SELECT SUM(a.cant_creditos) * rc.valorCredito INTO v_total FROM volante_matricula_asignatura vma JOIN asignatura a ON vma.id_asignatura = a.id_asignatura JOIN volante_matricula v ON vma.id_volante = v.id_volante JOIN regla_cobro rc ON (v.id_programa = rc.id_programa AND v.id_periodo = rc.id_periodo AND v.modalidad = rc.modalidad) WHERE vma.id_volante = v_id_volante; UPDATE volante_matricula SET monto_total = NVL(v_total, 0) WHERE id_volante = v_id_volante; END;
trg_generar_movimiento_cobro Importante

Al confirmar un volante (cambiar estado a PENDIENTE), se debe generar automáticamente el movimiento de cobro en la cuenta corriente del estudiante.

CREATE OR REPLACE TRIGGER trg_generar_movimiento_cobro AFTER UPDATE OF estado ON volante_matricula FOR EACH ROW WHEN (NEW.estado = 'PENDIENTE' AND OLD.estado != 'PENDIENTE') DECLARE v_id_cuenta NUMBER; v_cod_cobro VARCHAR2(20); BEGIN SELECT id_cuenta INTO v_id_cuenta FROM cuenta_corriente WHERE id_estudiante = :NEW.id_estudiante; v_cod_cobro := CASE :NEW.modalidad WHEN 'GLOBAL' THEN 'PMAT' WHEN 'CREDITOS' THEN 'PCRE' END; INSERT INTO movimiento (id_mov, fecha, valor, descripcion, codigo_detalle, id_cuenta, id_volante, id_periodo) VALUES (seq_movimiento.NEXTVAL, SYSDATE, :NEW.monto_total, 'Cobro matrícula periodo ' || :NEW.id_periodo, v_cod_cobro, v_id_cuenta, :NEW.id_volante, :NEW.id_periodo); END;

08
Módulos de la página web
Todo lo que el enunciado exige que la aplicación gestione, organizado por módulo.
Módulo 1 — Gestión de maestros SUPERVISOR + ADMIN

Datos de configuración del sistema. Deben ingresarse antes de poder operar.

  • CRUD de programas académicos
  • CRUD de asignaturas y asignación al plan de estudio por semestre
  • CRUD de períodos académicos (nombre, fecha inicio, fecha fin)
  • CRUD de reglas de cobro — por programa, período y modalidad, ingresando valorGlobal y valorCredito
  • CRUD de códigos de detalle con su grupo (COBRO/PAGO)
Módulo 2 — Gestión de estudiantes SUPERVISOR + ADMIN
  • CRUD completo del estudiante (nombre, apellido, carnet, correo, teléfono)
  • Asignación del programa académico al estudiante
Módulo 3 — Generación de cobros ASISTENTE + ADMIN

La funcionalidad central del sistema según el enunciado.

  • Individual: seleccionar estudiante → seleccionar período → ingresar semestre → seleccionar modalidad
  • Si modalidad = CREDITOS: mostrar lista de asignaturas del semestre para que el usuario seleccione → calcular monto con cant_creditos × valorCredito
  • Si modalidad = GLOBAL: aplicar valorGlobal de la regla de cobro directamente
  • Masiva: seleccionar período y programa → generar volantes para todos los estudiantes de ese programa
  • La generación debe crear la cuenta corriente si el estudiante no la tiene (el trigger lo hace automáticamente)
Módulo 4 — Cuenta corriente ASISTENTE + ADMIN

Consulta por estudiante y período. Usar la vista v_cuenta_corriente_detalle.

  • Mostrar cada movimiento con: código de detalle, descripción breve, valor, fecha
  • Mostrar saldo acumulado después de cada movimiento
  • Mostrar balance final del período: COBROs – PAGOs = saldo (debe tender a 0)
  • Mostrar resumen del volante asociado al período
Módulo 5 — Registro de pagos ASISTENTE + ADMIN
  • Seleccionar estudiante y período activo
  • Ingresar: medio de pago, referencia, valor pagado, fecha
  • El sistema crea la transaccion_pago y el movimiento de tipo PAGO (código MPAG) en la cuenta corriente
  • Actualizar automáticamente el estado del volante si el pago cubre el monto total
Módulo 6 — Administración del sistema Solo ADMIN
  • CRUD de usuarios — crear usuario, asignar persona y perfil
  • Enviar contraseña al correo de la persona asignada
  • Gestión de perfiles, permisos y menús
  • Asignación de permisos a perfiles

Reportes
Los 5 reportes exactos que el enunciado exige. Cada uno necesita su propia vista o consulta.
Reporte 1 — Listado general de estudiantesv_rpt_listado_general

Columnas requeridas: estudiante, programa, modalidad de cobro, monto/valor del volante. Filtrable por período.

CREATE OR REPLACE VIEW v_rpt_listado_general AS SELECT e.nombre||' '||e.apellido AS estudiante, p.nombre_programa, v.modalidad, v.monto_total, v.estado, per.nombre_periodo FROM volante_matricula v JOIN estudiante e ON v.id_estudiante = e.id_estudiante JOIN programa_academico p ON v.id_programa = p.id_programa JOIN periodo_academico per ON v.id_periodo = per.id_periodo;
Reporte 2 — Ingreso esperado por período y programav_rpt_ingreso_esperado

Totaliza el monto de todos los volantes generados por período y programa. Es la proyección de ingresos.

CREATE OR REPLACE VIEW v_rpt_ingreso_esperado AS SELECT per.nombre_periodo, p.nombre_programa, COUNT(*) AS num_estudiantes, SUM(v.monto_total) AS total_esperado FROM volante_matricula v JOIN programa_academico p ON v.id_programa = p.id_programa JOIN periodo_academico per ON v.id_periodo = per.id_periodo GROUP BY per.nombre_periodo, p.nombre_programa;
Reporte 3 — Pendientes de pago por programav_rpt_pendientes

El enunciado especifica que debe solicitar o escoger de una lista el programa académico antes de mostrar el resultado. La vista filtra por estado.

CREATE OR REPLACE VIEW v_rpt_pendientes AS SELECT e.nombre||' '||e.apellido AS estudiante, p.nombre_programa, per.nombre_periodo, v.monto_total, sp.total_pagos, v.monto_total - sp.total_pagos AS saldo_pendiente FROM volante_matricula v JOIN estudiante e ON v.id_estudiante = e.id_estudiante JOIN programa_academico p ON v.id_programa = p.id_programa JOIN periodo_academico per ON v.id_periodo = per.id_periodo JOIN v_saldo_periodo sp ON (sp.id_estudiante = e.id_estudiante AND sp.id_periodo = v.id_periodo) WHERE v.estado != 'PAGADO'; -- La app filtra por id_programa al consultar esta vista
Reporte 4 — Ingreso real recibidov_rpt_ingreso_real

Solo suma los movimientos de tipo PAGO (grupo = 'PAGO'). Es lo que realmente entró a caja en el período.

CREATE OR REPLACE VIEW v_rpt_ingreso_real AS SELECT per.nombre_periodo, p.nombre_programa, SUM(m.valor) AS total_recibido FROM movimiento m JOIN codigo_detalle cd ON m.codigo_detalle = cd.codigo_Detalle JOIN cuenta_corriente cc ON m.id_cuenta = cc.id_cuenta JOIN estudiante e ON cc.id_estudiante = e.id_estudiante JOIN programa_academico p ON e.id_programa = p.id_programa JOIN periodo_academico per ON m.id_periodo = per.id_periodo WHERE cd.grupo = 'PAGO' GROUP BY per.nombre_periodo, p.nombre_programa;
Reporte 5 — Estudiantes con crédito financierov_rpt_credito

Muestra estudiantes que tienen el código CRED en sus movimientos. Totaliza la cartera por cobrar.

CREATE OR REPLACE VIEW v_rpt_credito AS SELECT e.nombre||' '||e.apellido AS estudiante, p.nombre_programa, per.nombre_periodo, SUM(m.valor) AS valor_credito FROM movimiento m JOIN cuenta_corriente cc ON m.id_cuenta = cc.id_cuenta JOIN estudiante e ON cc.id_estudiante = e.id_estudiante JOIN programa_academico p ON e.id_programa = p.id_programa JOIN periodo_academico per ON m.id_periodo = per.id_periodo WHERE m.codigo_detalle = 'CRED' GROUP BY e.nombre, e.apellido, p.nombre_programa, per.nombre_periodo; -- UNION ALL con la suma total al final en la aplicación

Orden de ejecución del script DDL
El orden es estricto en Oracle. Una FK que referencia una tabla que no existe falla.
  • 1Secuencias — todas primero, no dependen de nada
  • 2Tablas independientes — perfil, menu, persona, programa_academico, periodo_academico, codigo_detalle, asignatura
  • 3Tablas de primera dependencia — permiso (→menu), usuario (→perfil, persona), regla_cobro (→programa, periodo), plan_estudio (→programa), estudiante (→programa)
  • 4Tablas puente simples — perfil_permiso, plan_estudio_asignatura
  • 5Tablas de segunda dependencia — cuenta_corriente (→estudiante), volante_matricula (→estudiante, periodo, regla_cobro)
  • 6Tablas de tercera dependencia — movimiento (→cuenta, volante, periodo, codigo_detalle), volante_matricula_asignatura
  • 7transaccion_pago — depende de movimiento, va al final
  • 8Índices — después de todas las tablas
  • 9Vistas — después de tablas e índices
  • 10Triggers — después de todo lo demás
  • 11INSERT de datos iniciales — códigos de detalle, perfiles, primer usuario admin
